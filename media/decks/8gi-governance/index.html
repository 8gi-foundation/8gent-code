Four-Tier Membership

Clear roles, clear expectations, clear boundaries.

Founder

1 member - James Spalding
  • Veto on constitutional decisions
  • Standard vote on operational
  • Trademark and IP holder
  • Final arbiter on disputes

Core

3-12 members
  • Full voting rights
  • Requires 6+ months + 10+ PRs
  • Merge rights on repos
  • 10% referral revenue share

Contributor

Unlimited
  • Advisory vote (non-binding)
  • Nominated by Core member
  • 5% referral revenue share
  • 30-day probation period

Observer

Unlimited
  • No vote
  • Read-only access
  • Any member can invite
  • No formal approval needed

Invitation & Vetting

Higher access requires higher trust. Each tier has its own gate.

1

Observer

Any member can invite. No formal approval required. Instant access to read-only channels and docs.

2

Contributor

48-hour objection window after nomination. If no Core member objects, approved automatically. 30-day probation starts on acceptance.

3

Core

Founder approval required. Must demonstrate sustained track record: 6+ months active, 10+ merged PRs, consistent quality. No shortcuts.

4

Founder

Non-transferable. There is one Founder. This is a constitutional constant.

Decision Categories

Different decisions require different levels of consensus.

Category Who Votes Threshold Voting Window Founder Role
Constitutional Founder + Core 75% supermajority 7 days Veto power
Strategic Founder + Core 51% majority 7 days Tiebreak
Operational Core only 51% majority 72 hours No special role
Community All members Advisory only 72 hours Standard vote

Voting mechanism: Async in Telegram. Each vote is a reply to the proposal thread. Silence after the window closes counts as abstention, not approval.

Code Review at Scale

The review process evolves as the circle grows.

1
Phase 1: Seed NOW
1-5 members. Founder reviews all PRs. High trust, personal vetting. Direct feedback loop.
2
Phase 2: Growth NEXT
5-15 members. Founder + Core review together. Shared responsibility, faster throughput.
3
Phase 3: Distributed LATER
15-50 members. Core handles most reviews. Founder focuses on architecture-level PRs only.
4
Phase 4: Domain Teams SCALE
50+ members. Domain-specific maintainers own their areas. Founder reviews cross-domain changes.

Quarantine Branch Pipeline

1. Push to quarantine/*
2. CI runs automated checks
3. Auto-generates PR from quarantine
4. Human review (per phase rules)
5. Merge to main

No direct pushes to main. Ever. Quarantine is the only path.

Dispute Resolution & Removal

Process protects everyone. Including the circle from bad actors.

3-Step Resolution

1

Informal Discussion

Direct conversation between parties. Most issues resolve here.

2

Mediation (14 days)

Core member mediates. Written record. Both sides heard.

3

Founder Adjudication (7 days)

Final decision. Binding. Published to the circle.

Removal Process

Grounds for Removal

  • Constitutional violation
  • Inactivity: 6 months (Core), 12 months (Contributor)
  • Security breach or data mishandling
  • Sustained pattern of harmful behavior

Process

  • Written warning with 14-day response window
  • Core vote: 51% for Contributor, 75% for Core
  • 24-hour access revocation after vote
  • Emergency: Founder can suspend immediately, ratified by Core within 7 days

Scaling Phases

The governance structure evolves with the community.

●
Seed NOW 1-5 members
High-trust, personal vetting. Founder knows every member. Decisions made in real-time conversation. Move fast, stay small.
●
Growth NEXT 5-15 members
Distributed code review. Quarterly sync meetings. Core members share review load. Contributor pipeline established.
●
Scale LATER 15-50 members
Sub-teams form around domains (kernel, TUI, infrastructure). Elected team leads. Monthly all-hands. Written RFCs for strategic changes.
●
Network FUTURE 50-200 members
Federated structure. Regional circles with local autonomy. Cross-circle coordination through elected representatives.
○
Movement VISION 200-500+ members
Constitutional democracy. Elected council replaces Founder tiebreak on operational decisions. Founder retains constitutional veto only.

Emergency Powers

Speed when it matters. Accountability always.

Founder Unilateral Action

The Founder can act without prior vote in three specific scenarios:

  • Security vulnerabilities - active exploits, credential leaks, supply chain attacks
  • Member violations - immediate harm to users, data breaches, malicious code
  • Infrastructure failure - service outages requiring immediate architectural changes

Safeguards

  • All emergency actions must be ratified by Core within 7 days
  • Every action is logged with timestamp, rationale, and impact
  • Logs are visible to all Core members
  • If Core does not ratify, the action must be reversed

Transparency Principle

Emergency powers exist for speed, not secrecy. The circle sees everything. Trust is maintained through radical transparency, not authority.
