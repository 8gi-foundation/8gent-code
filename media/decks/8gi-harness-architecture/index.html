Monoliths Break Developers

Why the current generation of agent tools is a dead end

Single Trust Boundary

One compromised tool means all tools are compromised. There is no way to grant "read my code" without also granting "delete my code."

Global Configuration

You cannot give your work project strict policies and your hobby project relaxed ones. The agent treats every context the same.

Binary Scaling

You are either running locally (free, limited) or paying for cloud (powerful, expensive). There is no middle ground.

All or Nothing

Every agent tool ships as a monolith. You get everything or nothing. A file-read tool bundles the same permissions as a production deploy tool.

The result: Developers either over-trust a black box or abandon the tool entirely.

The Harness

A lean kernel that does three things and nothing else

Agent Loop input / stream / execute
Tool Proxy delegate / gate
Model Router local / cloud / hybrid

Agent Loop

Take user input, stream to LLM, execute tool calls, return output. Powered by Vercel AI SDK.

packages/eight/agent.ts

Tool Proxy

Every tool call passes through a single gateway. The harness does not execute tools directly. It delegates to the Tool Belt.

packages/ai/toolshed-bridge.ts

Model Router

Local Ollama, OpenRouter free, or any provider. Task router selects the right model. Fast tasks get small models. Complex tasks get capable ones.

packages/providers/

What the Harness Does NOT Do

Store data. Manage permissions. Register tools. Handle billing. Know about other users. It is stateless by design.

The Tool Belt

Your tools. Your rules. Per-user, per-project, capability-gated.

Capability Discovery

Agents find tools by capability, not by name. Only matching tools load into context. Token-efficient summaries (~50 tokens) before full load.

packages/toolshed/registry/discovery.ts

NemoClaw Policy Engine

YAML-based, deny-by-default. 15 default policies ship with install. Three decisions: BLOCK APPROVE ALLOW

packages/permissions/policy-engine.ts

28 Tools, 5 Categories

  • Code: outline, symbol, search (4 tools)
  • File: read, write, edit, list (4 tools)
  • Git: status, diff, log, add, commit (5 tools)
  • Execution: run, spawn, check agents (4 tools)
  • Web, Design, Browser, Memory (11 tools)

User Override

Your policies live at ~/.8gent/policies.yaml. Plain YAML. Human-readable conditions. Edit it, own it.

Secrets in files - blocked. Force push - blocked. Pipe to shell - requires approval. Hard reset - requires approval.

The Tool Shed

Collective intelligence without collective data

Tool Registry

Contributors register tools. The registry organizes by domain: code, design, execution, github, repo. 8 module files extracted across 5 categories today.

packages/toolshed/registry/index.ts

Skill Registry

Full skill data stays on disk. Only summaries (~50 tokens) kept in memory. Load on invocation, not on boot. Keeps agent context lean.

packages/toolshed/skill-registry.ts

Collective Patterns

When 100 developers solve similar problems, the successful patterns surface. Not the code. Not the data. Just the abstracted approach.

"Most developers who fixed this type of TypeScript error used this sequence of tools." That is collective intelligence without surveillance.

Language Agnostic

The Shed API speaks JSON over HTTP. Python tools, Rust tools, Go tools - they all register the same way. The Harness does not care what language a tool is written in.
