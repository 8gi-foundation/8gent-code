## 1. Purpose

This document defines the security framework for 8GI (Infinite General Intelligence) - a collective intelligence where N humans run AI agents that generate code and submit pull requests to shared codebases. This architecture creates attack surface that scales with every participant. This framework exists to keep that surface auditable, bounded, and recoverable.

Every member of the 8GI circle, and every agent operating on their behalf, is bound by this framework. Violations result in access revocation per Section 8.

---

## 2. Threat Model

### 2.1 Attack Surface

8GI's threat model differs from traditional open source. The code is not written by humans alone - it is generated by LLMs operating with varying degrees of autonomy. Each participant introduces:

- **An AI agent** with filesystem access, command execution, git operations, and network capability
- **An LLM backend** (local or cloud) that may hallucinate, inject, or reproduce vulnerable patterns
- **A human operator** who may misconfigure, over-trust, or act with malicious intent

### 2.2 Threat Categories

| ID | Threat | Vector | Severity |
|----|--------|--------|----------|
| T1 | **Malicious code injection** | Compromised member's agent submits backdoored PR | Critical |
| T2 | **Supply chain poisoning** | LLM generates code that imports malicious dependencies | Critical |
| T3 | **Secret exfiltration** | Agent reads .env or credentials and leaks via network/commit | Critical |
| T4 | **Privilege escalation** | Agent bypasses NemoClaw policy to execute blocked commands | High |
| T5 | **Data poisoning** | Adversarial member corrupts shared training data or memory | High |
| T6 | **Social engineering** | Member tricks another member's agent via shared context | High |
| T7 | **Dependency confusion** | LLM suggests internal package name that resolves to public malicious package | High |
| T8 | **Prompt injection** | Malicious content in PRs or issues that hijacks reviewing agents | Medium |
| T9 | **Resource exhaustion** | Agent runs infinite loops, fills disk, or spawns unbounded processes | Medium |
| T10 | **Stale credential exposure** | Rotated secrets remain in git history or logs | Medium |
| T11 | **Model supply chain** | Compromised local model (Ollama) generates targeted vulnerabilities | Medium |
| T12 | **Cross-member contamination** | One member's patterns or errors propagate to others via shared pipelines | Low |

### 2.3 Trust Boundaries

```
UNTRUSTED                    QUARANTINE                   TRUSTED
-----------                  -----------                  -----------
LLM output                   quarantine/* branches        main branch
External deps                PR review queue              Released packages
New member agents            CI pipeline                  Signed releases
Network responses            Staging environment          Production deploys
```

All LLM-generated code starts untrusted. It crosses into quarantine via commit. It crosses into trusted only after passing CI and human review.

## 3. Supply Chain Security

### 3.1 LLM Output is Untrusted Input

All code generated by any LLM - local or cloud - is treated as untrusted input. This is not optional. LLMs reproduce vulnerable patterns from training data, hallucinate nonexistent APIs, and can be manipulated via prompt injection.

### 3.2 Dependency Controls

| Control | Implementation |
|---------|---------------|
| **Lock files mandatory** | `bun.lockb` must be committed. PRs that modify it trigger additional review. |
| **No install scripts** | `package.json` must not contain `preinstall`, `postinstall`, or `prepare` scripts that execute arbitrary code without review. |
| **Allowlisted registries** | Only `npmjs.com` and `jsr.io`. No custom registries without CSO approval. |
| **Dependency audit** | `bun audit` (or equivalent) runs in CI. Any `critical` or `high` vulnerability blocks merge. |
| **New dependency review** | Any PR that adds a new dependency requires explicit justification and human approval. |
| **Typosquat detection** | CI checks new package names against known typosquat patterns for existing dependencies. |

### 3.3 Model Supply Chain

- Local models (Ollama) must be pulled from official Ollama library only
- Model checksums are verified on first pull
- Custom fine-tuned models used in the kernel pipeline (`packages/kernel/`) must be signed by the CAO before promotion into the model router
- OpenRouter usage is opt-in and disclosed per the 8GI Constitution

## 4. NemoClaw Policy Template for Circle Members

Circle members run with a **stricter** NemoClaw policy than the default shipped with 8gent Code. The following YAML is deployed via `8gi-setup` and loaded from `~/.8gent/policies.yaml` on every member machine.

```yaml
version: 1

# 8GI Circle Member Policy - STRICT MODE
# This policy overrides default-policies.yaml for circle contributors.
# Infinite mode is DISABLED. All destructive actions require approval.

policies:
  # -- Secrets: hard block, no exceptions --

  - name: "8gi-no-secrets-in-code"
    action: write_file
    condition: >
      content contains API_KEY or content contains SECRET or
      content contains PASSWORD or content contains TOKEN or
      content contains PRIVATE_KEY or content contains aws_access or
      content contains ssh-rsa or content contains BEGIN RSA or
      content contains BEGIN OPENSSH or content contains Bearer
    decision: block
    message: "Secrets in code are blocked. Use environment variables via .env (gitignored)."

  - name: "8gi-no-env-write"
    action: write_file
    condition: "path ends_with .env or path ends_with .env.local or path ends_with .env.production"
    decision: block
    message: "Circle agents cannot write .env files. Secrets are managed manually by the human operator."

  # -- Filesystem: restrict blast radius --

  - name: "8gi-no-write-outside-project"
    action: write_file
    condition: "path contains /usr or path contains /etc or path contains /System or path starts_with /tmp"
    decision: block
    message: "Write outside project directory is blocked for circle contributions."

  - name: "8gi-no-delete-critical"
    action: delete_file
    condition: >
      path contains package.json or path contains tsconfig or
      path contains CLAUDE.md or path contains bun.lockb or
      path contains .github or path contains policies.yaml
    decision: block
    message: "Deletion of critical project files is blocked."

  # -- Commands: approval required for anything beyond safe list --

  - name: "8gi-allow-safe-commands"
    action: run_command
    condition: >
      command starts_with git status or command starts_with git diff or
      command starts_with git log or command starts_with git branch or
      command starts_with ls or command starts_with cat or
      command starts_with bun run or command starts_with bun test or
      command starts_with tsc or command starts_with npx tsc or
      command starts_with echo
    decision: allow
    message: "Safe read-only and build commands are allowed."

  - name: "8gi-block-destructive"
    action: run_command
    condition: >
      command contains rm -rf or command contains sudo or
      command contains chmod 777 or command contains mkfs or
      command contains dd if= or command contains :(){ :|: or
      command contains curl|sh or command contains wget|sh
    decision: block
    message: "Destructive and dangerous commands are blocked for circle members."

  - name: "8gi-approve-other-commands"
    action: run_command
    condition: "command contains /"
    decision: require_approval
    message: "Non-allowlisted commands require human approval."

  # -- Git: quarantine branch enforcement --

  - name: "8gi-no-push-main"
    action: git_push
    condition: "branch in [main, master, production, prod, release]"
    decision: block
    message: "Circle members cannot push directly to protected branches. Use quarantine/* branches."

  - name: "8gi-no-force-push"
    action: run_command
    condition: "command contains git push --force or command contains git push -f"
    decision: block
    message: "Force push is blocked for all circle members."

  - name: "8gi-no-hard-reset"
    action: run_command
    condition: "command contains git reset --hard or command contains git checkout ."
    decision: block
    message: "Hard reset and bulk checkout are blocked. Use targeted reverts."

  # -- Network: restrict exfiltration vectors --

  - name: "8gi-block-exfil"
    action: network_request
    condition: >
      url contains pastebin.com or url contains ngrok.io or
      url contains requestbin or url contains webhook.site or
      url contains pipedream.com or url contains hookbin
    decision: block
    message: "Requests to known exfiltration endpoints are blocked."

  - name: "8gi-approve-network"
    action: network_request
    condition: "url contains http"
    decision: require_approval
    message: "Network requests require human approval for circle contributions."

  # -- Desktop: blocked entirely for circle contributions --

  - name: "8gi-no-desktop"
    action: desktop_use
    condition: "action contains click or action contains type or action contains press"
    decision: block
    message: "Desktop automation is disabled for circle contributions."
```

### 4.1 Policy Enforcement

- Policies are loaded at agent startup via `loadPolicies()` in `packages/permissions/policy-engine.ts`
- The `8gi-setup` script deploys the circle policy to `~/.8gent/policies.yaml`
- Policies cannot be disabled by the agent - only the human operator can edit the YAML
- The policy file itself is integrity-checked via SHA-256 hash stored in `~/.8gent/policy-checksum`
- If the checksum fails on startup, the agent refuses to operate until the human re-verifies

## 5. Required CI Checks

No PR merges to `main` without ALL of the following passing. These are configured as required status checks in GitHub branch protection rules.

### 5.1 Automated Gate

| Check | Tool | Blocks Merge |
|-------|------|:------------:|
| **TypeScript compilation** | `tsc --noEmit` | Yes |
| **Lint** | `eslint` with 8gent config | Yes |
| **Unit tests** | `bun test` | Yes |
| **Dependency audit** | `bun audit` or `npm audit` | Yes (critical/high) |
| **Secret scan** | `trufflehog` or `gitleaks` on diff | Yes |
| **Policy validation** | NemoClaw dry-run on changed files | Yes |
| **Line count check** | No single file addition exceeds 200 lines | Yes |
| **Lock file integrity** | `bun.lockb` matches `package.json` | Yes |
| **CHANGELOG entry** | PR includes CHANGELOG.md update | Yes |

### 5.2 Human Gate

| Requirement | Who | Blocks Merge |
|-------------|-----|:------------:|
| **Code review** | At least 1 approved review from trusted engineer | Yes |
| **Security review** | CSO or designated security reviewer for PRs touching `packages/permissions/`, `.github/`, `packages/daemon/`, or any policy file | Yes |
| **CAO final approval** | James Spalding for PRs from new circle members (first 10 PRs) | Yes |

### 5.3 Post-Merge

- Automated benchmark regression check runs on `main` after merge
- If benchmarks regress beyond threshold, an automatic revert PR is opened
- Release builds trigger checksum generation (Section 11)

## 6. Secret Management

### 6.1 Rules

1. **No secrets in code. Ever.** The NemoClaw policy hard-blocks any write containing secret patterns (API_KEY, TOKEN, PASSWORD, PRIVATE_KEY, Bearer tokens, SSH keys).
2. **No secrets in git history.** If a secret is accidentally committed, the incident response process (Section 8) is triggered immediately. The secret is rotated, and `git filter-branch` or `BFG Repo Cleaner` removes it from history.
3. **`.env` files are gitignored.** The `.gitignore` at repo root must contain `.env*`. CI verifies this.
4. **Circle members manage their own secrets locally.** The `~/.8gent/` directory stores local config. Agents cannot read or write `.env` files (blocked by circle policy).
5. **Shared secrets (CI, deployment)** are stored in GitHub Actions secrets or the deployment platform's secret manager (Fly.io secrets). Never in code, config files, or documentation.

### 6.2 Secret Scanning

- `trufflehog` or `gitleaks` runs on every PR diff
- A pre-commit hook runs the same scan locally before commits leave the machine
- The scan checks for: API keys, tokens, passwords, private keys, connection strings, AWS credentials, GCP service account JSON, and high-entropy strings matching known patterns

### 6.3 Rotation Policy

| Secret Type | Rotation Frequency | Owner |
|-------------|-------------------|-------|
| GitHub tokens | 90 days | Individual member |
| API keys (OpenRouter, etc.) | 90 days | Individual member |
| Deployment secrets (Fly.io) | 180 days | CAO |
| npm publish tokens | 180 days | CAO |
| SSH keys | Annual | Individual member |

## 7. Quarantine System

### 7.1 Branch Naming

All LLM-generated code from circle members must be committed to quarantine branches:

```
quarantine/<member-username>/<description>
```

Examples:
- `quarantine/alice/add-cache-utility`
- `quarantine/bob/fix-memory-leak`

### 7.2 Flow

```
LLM generates code
       |
       v
Agent commits to quarantine/<member>/<feature>
       |
       v
Agent pushes to remote quarantine branch
       |
       v
Agent opens PR from quarantine branch -> main
       |
       v
CI runs all automated checks (Section 5.1)
       |
       v
Human review (Section 5.2)
       |
       v
Merge to main (squash merge only)
       |
       v
Quarantine branch deleted
```

### 7.3 Rules

- Circle member agents CANNOT push directly to `main`, `master`, `production`, `prod`, or `release` (enforced by NemoClaw policy and GitHub branch protection)
- Quarantine branches are ephemeral - deleted after merge or after 30 days of inactivity
- No quarantine branch may contain more than 500 lines of changes (enforced by CI)
- Quarantine branches from the same member cannot exceed 3 open PRs at a time to prevent review fatigue and maintain audit quality

### 7.4 Emergency Bypass

In the event of a critical production issue requiring immediate fix:
1. The CAO (James) may push directly to `main`
2. A post-hoc review must be completed within 24 hours
3. The bypass is logged in the incident log with justification

## 8. Audit Trail Requirements

### 8.1 What Gets Logged

Every agent session produces an audit log. The following events are captured:

| Event | Data Logged |
|-------|-------------|
| **Tool call** | Tool name, arguments (secrets redacted), timestamp, policy decision |
| **File write** | File path, line count, SHA-256 of content |
| **File delete** | File path, SHA-256 of deleted content |
| **Command execution** | Full command (secrets redacted), exit code, duration |
| **Git operation** | Operation type, branch, commit SHA |
| **Network request** | URL (query params redacted), method, status code |
| **Policy evaluation** | Rule matched, decision, context fields |
| **Error** | Error type, message, stack trace (no secrets) |

### 8.2 Log Storage

- **Local:** `~/.8gent/audit/` - JSON Lines format, one file per session
- **Retention:** 90 days minimum on local machine
- **Format:** Each line is a JSON object with `timestamp`, `event`, `data`, `sessionId`, `memberId`
- **Integrity:** Each log file has a SHA-256 checksum appended on session close

### 8.3 Log Access

- Members can review their own audit logs at any time
- The CSO may request audit logs from any member during an incident investigation (Section 8.4)
- Logs are never uploaded automatically - they are shared only on request during investigations
- Logs must NOT contain secret values - the logging layer redacts patterns matching the NemoClaw secret detection rules before writing

## 9. Incident Response

### 9.1 Classification

| Severity | Definition | Response Time |
|----------|-----------|---------------|
| **P0 - Critical** | Secret leaked to public, malicious code merged to main, production compromised | Immediate (within 1 hour) |
| **P1 - High** | Suspicious PR detected, policy bypass attempted, agent acting outside scope | Within 4 hours |
| **P2 - Medium** | Failed secret scan, unusual network activity, audit log anomaly | Within 24 hours |
| **P3 - Low** | Policy misconfiguration, minor compliance gap, documentation issue | Within 1 week |

### 9.2 Rogue Agent Protocol

If a member's agent is behaving outside expected parameters:

1. **Kill:** The member immediately terminates the agent session (`Ctrl+C`, `kill`, or machine shutdown)
2. **Isolate:** Revoke the member's GitHub push access via org settings
3. **Preserve:** Copy audit logs from `~/.8gent/audit/` before any cleanup
4. **Assess:** CSO reviews audit logs to determine scope of compromise
5. **Contain:**
   - If code was pushed: revert all commits from the affected agent
   - If secrets were exposed: rotate all potentially affected credentials
   - If data was exfiltrated: assess what was sent and to where
6. **Remediate:** Fix the root cause (policy gap, misconfiguration, compromised model)
7. **Report:** Document the incident in `docs/incidents/YYYY-MM-DD-<description>.md`
8. **Review:** Determine if the member retains circle access

### 9.3 Member Removal

A member is removed from the circle if:
- They intentionally violate the Constitution
- Their agent repeatedly fails security checks without remediation
- They refuse to share audit logs during an investigation
- They attempt to bypass NemoClaw policies

Removal process:
1. Revoke GitHub org access
2. Revoke Telegram group access
3. Rotate any shared secrets the member had access to
4. Document the removal reason (kept private, shared only with the member)

## 10. Privacy Enforcement

### 10.1 Principles

The 8GI Constitution states: "Personal data never leaves the user's machine without explicit opt-in." This section enforces that principle technically.

### 10.2 Controls

| Control | Implementation |
|---------|---------------|
| **No telemetry** | Zero analytics, zero tracking, zero phone-home. Verified by code review. |
| **Local-only memory** | `packages/memory/store.ts` writes to `~/.8gent/memory.db` (local SQLite). No cloud sync without opt-in. |
| **No PII in commits** | CI scans PR diffs for email addresses, phone numbers, physical addresses, and names using pattern detection. Flagged for review. |
| **No PII in logs** | Audit log redaction layer strips patterns matching emails, IPs, and credential formats before write. |
| **Anonymised patterns only** | The factory pipeline processes usage patterns (tool frequency, error categories, task types). Raw code, file contents, and personal data are never transmitted. |
| **Memory encryption** | SQLite memory store to be encrypted at rest (in progress, tracked in codebase). |
| **Right to delete** | Any member can run `8gent purge` to delete all local data, memory, and audit logs. |

### 10.3 Verification

- Quarterly audit: grep the codebase for outbound network calls and verify each one is documented and justified
- Any new network endpoint added to the codebase requires CSO sign-off
- The `packages/tools/browser/` module (DuckDuckGo scraper, fetch with cache) is the only approved outbound HTTP path for agent operations

## 11. The 200-Line Rule as Security

### 11.1 Rationale

The 8GI Constitution mandates: "No ability exceeds 200 lines." This is not merely a code quality rule - it is a security control.

### 11.2 Security Properties

| Property | How 200 Lines Achieves It |
|----------|--------------------------|
| **Auditability** | A 200-line file can be fully reviewed by a human in under 10 minutes. A 2000-line file cannot. |
| **Blast radius** | A compromised or buggy 200-line module affects one capability. A monolithic file affects everything. |
| **Diff clarity** | PRs touching 200-line files produce diffs that fit on one screen. Hidden changes are harder to sneak in. |
| **Dependency isolation** | Small modules import less, reducing transitive attack surface. |
| **Replacement cost** | If a 200-line module is compromised, it can be rewritten from scratch in one session. A 2000-line module cannot. |

### 11.3 Enforcement

- CI check: any new or modified file exceeding 200 lines of code (excluding comments and blank lines) fails the build
- Exceptions require CSO approval and must be documented with justification
- The count excludes: type definition files (`.d.ts`), generated files, test fixtures, and configuration

## 12. Code Signing and Checksums

### 12.1 Release Integrity

Every release published to npm or deployed to production must include integrity verification.

### 12.2 Requirements

| Artifact | Integrity Method |
|----------|-----------------|
| **npm package** | SHA-512 integrity hash in npm registry (automatic). Package tarball checksum published in GitHub release. |
| **Git release tag** | GPG-signed tag by the CAO or designated release manager. |
| **Fly.io deployment** | Deploy triggered only from signed release tags on `main`. |
| **CHANGELOG** | Every version entry includes the commit range and date. |

### 12.3 Release Process

```
1. Version bump in package.json, bin/8gent.ts, README.md
2. CHANGELOG.md entry added
3. PR created, CI passes, human review approved
4. Merge to main (squash)
5. git tag -s v<version> -m "Release v<version>"
6. git push --tags
7. npm publish (from CI, using org-scoped token)
8. GitHub Release created with:
   - Tarball checksum (SHA-256)
   - CHANGELOG excerpt
   - Link to full diff
9. Fly.io deployment triggered from tag
```

### 12.4 Verification

Anyone can verify a release:

```bash
# Verify git tag signature
git tag -v v1.2.0

# Verify npm package integrity
npm pack @podjamz/8gent-code --dry-run
sha256sum podjamz-8gent-code-1.2.0.tgz

# Compare against published checksum in GitHub release
```

## 13. Annual Security Review

### 13.1 Schedule

A comprehensive security review is conducted annually, with the next review due by **2027-03-26**.

### 13.2 Scope

| Area | Review Activity |
|------|----------------|
| **Threat model** | Reassess all threats in Section 2. Add new threats discovered during the year. |
| **NemoClaw policies** | Audit all rules. Remove stale rules. Add rules for new attack vectors. |
| **CI pipeline** | Verify all checks still run and block correctly. Test with intentional failures. |
| **Dependencies** | Full audit of all direct and transitive dependencies. Remove unused packages. |
| **Audit logs** | Sample review of logs from 3 random member sessions. Verify completeness. |
| **Access control** | Review all GitHub org members. Remove inactive members. Rotate shared secrets. |
| **Incident review** | Review all incidents from the past year. Identify patterns. Update controls. |
| **Privacy audit** | Verify no new outbound network calls were added without approval. Verify memory encryption status. |
| **200-line compliance** | Scan entire codebase for files exceeding 200 lines. Remediate or document exceptions. |
| **This document** | Update version, review date, and any sections that no longer reflect reality. |

### 13.3 Output

The annual review produces:
1. Updated `8GI-SECURITY.md` (this document)
2. An incident summary for the year (anonymised)
3. A remediation plan for any gaps found
4. Updated NemoClaw policies if needed

### 13.4 Interim Reviews

In addition to the annual review, an interim review is triggered by:
- Any P0 or P1 incident
- Addition of more than 5 new circle members in a quarter
- Major architectural changes to the agent system
- Changes to the NemoClaw policy engine itself
