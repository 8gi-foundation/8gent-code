## Slide 2 - The Threat Model
Our threat model is different from traditional open source. The code is not written by humans alone. It is generated by LLMs with varying degrees of autonomy. Every participant introduces three threat vectors. An AI agent with filesystem access, command execution, git operations, and network capability. An LLM backend that may hallucinate, inject, or reproduce vulnerable patterns. And a human operator who may misconfigure, over-trust, or act with malicious intent. We define three trust boundaries. Untrusted: LLM output, external dependencies, new member agents, network responses. Quarantine: branches, PR review queue, CI pipeline. Trusted: main branch, released packages, signed releases. All code starts untrusted. It crosses into trusted only after CI and human review.

## Slide 3 - Supply Chain Security
All LLM output is untrusted input. Full stop. For dependencies: lock files are mandatory. No install scripts. Only npm js dot com and jsr dot io registries. Dependency audit runs in CI. Any new dependency needs explicit justification. Typosquat detection is active. For models: local models only from the official Ollama library. Checksums verified on pull. Custom fine-tuned models must be signed by the CAO before promotion.

## Slide 4 - The Quarantine System
All LLM-generated code from circle members must go through quarantine branches. Named quarantine slash member slash feature. The flow is: generate, commit to quarantine, push, open PR, CI runs all checks, human review, squash merge to main, quarantine branch deleted. Three hard limits. Five hundred lines maximum per PR. Three open PRs maximum per member. Thirty day expiry on inactive branches. Circle agents cannot push to main. Enforced by NemoClaw and GitHub branch protection.

## Slide 5 - Incident Response
Four severity levels. P0 Critical: secret leaked, malicious merge, production compromised. Response within one hour. P1 High: suspicious PR, policy bypass, agent acting outside scope. Four hours. P2 Medium: failed scan, unusual network activity, audit log anomaly. Twenty-four hours. P3 Low: policy misconfiguration, compliance gap, documentation issue. One week. If an agent goes rogue: kill it. Isolate the member's access. Preserve audit logs. Assess the damage. Contain the blast radius. Remediate the root cause. Report and document. Review membership.

## Slide 6 - The 200-Line Rule as Security
The two hundred line rule is not just code quality. It is a security control. Five properties. Auditability: a two hundred line file can be fully reviewed by a human in under ten minutes. Blast radius: a compromised module affects one capability, not everything. Diff clarity: PRs fit on one screen. Hidden changes are harder to sneak in. Dependency isolation: small modules import less, reducing attack surface. Replacement cost: if a two hundred line module is compromised, you rewrite it from scratch in one session. A two thousand line module? You cannot. Simplicity is security.
